# 鼠标手写签名生成器 — 产品与技术需求文档

> 交付对象：Codex / AI 编码助手
> 版本：v1.0

## 1. 背景与目标

用户没有触控板或手绘板，仅用鼠标无法画出美观的手写签名。本项目要实现一个本地桌面工具：用户输入姓名（中文和/或英文），程序自动在屏幕上任意位置、任意软件窗口内，用真实的鼠标按下-移动-松开动作，画出一段接近真人手写风格的签名笔迹（有笔锋、连笔、自然的速度变化），而不是机械的等速直线轨迹。

最终效果不是生成一张签名图片贴上去，而是真正驱动操作系统鼠标，在用户当前打开的任意应用（PDF阅读器、画图软件、网页签字控件等）的画布区域上"画"出签名。

## 2. 核心需求（必须满足）

1. **中英文双语支持**：用户可以输入中文姓名（如"张三"）或英文姓名（如"John Smith"），系统都能生成对应的手写笔迹轨迹。
2. **跨平台**：必须同时支持 macOS（Apple Silicon，用户机器为 M2 Max）和 Windows，鼠标控制层需要做平台适配。
3. **真实手写质感**：轨迹必须体现笔锋（起笔/收笔的轻重变化）、连笔（部分笔画之间不提笔直接划过去）、自然的运笔速度变化（直线快、转弯慢），而不是简单的等速折线。**这是本项目最核心也是难度最高的要求，中文手写质量不可妥协。**
4. **可在任意屏幕位置运行**：运行前用户框选或点击指定一块矩形区域作为"签字区"，程序将签名轨迹自动缩放、定位、对齐到这块区域内再执行鼠标动作。
5. **可重复使用、可配置**：同一个名字可以多次签出，允许有轻微随机变化（不要求每次像素级一致，更像真人会有细微差异），并支持调整签名大小、倾斜角度、书写速度。

## 3. 技术难点与关键设计决策

### 3.1 为什么不能直接用字体轮廓

普通字体（包括手写体字体）里的字形数据是"轮廓"（outline/glyph contour），也就是字的边界线，不是"笔尖运动的中心线路径"。如果直接把轮廓当作鼠标轨迹去画，效果会是先画出字的边框，类似空心字描边，而不是真实书写时笔尖划过的那条线。**必须避免这个误区。**

### 3.2 英文签名笔迹方案

英文（含姓名拼音）有现成的开源单线笔迹字体（Single-line / Skeleton font），这类字体本身存储的就是"一笔写成"的中心线路径，专为绘图笔、激光雕刻、CNC刻字等场景设计，可以直接当作运笔轨迹使用。建议技术路线：

- 使用 Hershey Fonts（开源单线矢量字体，有现成的 JSON/SVG 格式移植版本，可在 npm 或 PyPI 搜索 `hershey-fonts` / `svg-hershey-text`）或同类单线手写字体作为基础轨迹来源。
- 对原始关键点做贝塞尔曲线插值平滑处理，避免折线感。
- 字母之间的连笔：分析相邻字母起止点距离，距离小于阈值时不提笔直接连接（模拟连笔签名），距离大的字母间提笔重新落笔。

### 3.3 中文签名笔迹方案（核心难点，质量不可妥协）

中文没有现成的高质量"单线手写笔迹字体"覆盖常用汉字，需要使用真实采集的笔迹轨迹数据库。推荐方案如下，按优先级排列：

#### 方案一（首选）：CASIA-OLHWDB 在线手写数据库

- 中国科学院自动化研究所（CASIA / NLPR）发布的公开学术数据集，包含约 3,755 个常用汉字类别、超过 110 万笔迹样本，由 300 名书写者用电子笔实时采集，记录的是真实的笔尖运动顺序（pen-down/pen-up 状态 + 时间序列坐标），不是轮廓。
- 申请地址：搜索 "CASIA Online Handwriting Database" 或访问中科院自动化所模式识别国家重点实验室官网数据库页面，需要按学术用途申请下载（通常免费，需邮箱注册）。
- 数据格式为 `.pot` / `.gnt` 等二进制格式，需要编写解析脚本（开源社区已有较成熟的 Python 解析器可参考，需要 Codex 在实现时搜索确认最新可用的解析代码）。
- 实现思路：为常用姓氏和名字用字预先解析、缓存若干个真实笔迹样本（建议覆盖百家姓常见字 + 常用名字用字，做成本地笔迹库，比如 JSON 格式存储每个字若干笔迹样本的坐标序列），用户输入姓名时直接拼接对应汉字的真实笔迹轨迹，并在笔画之间做平滑过渡和整体连笔处理（比如字与字之间用一条衔接曲线连起来，模拟签名连笔）。

#### 方案二（备选/兜底）：开源手写体生成项目

如果 CASIA 数据集申请或解析遇到阻碍，可以考虑以下方向作为笔迹数据或生成模型来源，实现前需要先用 web 搜索确认这些项目当前的可用性、License 和效果：

- 搜索关键词：`Chinese handwriting generation stroke sequence open source`、`手写体生成 笔迹数据 开源`、`RNN handwriting synthesis Chinese`。
- 可关注是否有基于 Graves handwriting synthesis（Alex Graves 2013 论文思路）训练的中文笔迹生成模型开源实现，这类模型输出的就是笔迹的坐标序列 + 抬笔信号，天然适合本项目。
- 也可以考虑用书法字体（如田英章楷书等）的轮廓做"骨架提取"（skeletonization，比如用 OpenCV 的细化算法 `cv2.ximgproc.thinning`）将轮廓变成中心线，再转成轨迹点，效果不如真实笔迹库但实现门槛更低，可作为过渡方案先跑通整体流程。

> ⚠️ 中文方案的实现难度明显高于英文，建议 Codex 按"里程碑"分阶段开发：先用方案二的骨架提取法或简化笔迹打通鼠标自动化全流程，验证可行性后，再替换为方案一的真实笔迹库以提升质量。两个方案的下游接口（统一输出"轨迹点序列+抬笔标记"的数据结构）应保持一致，方便后期切换数据源。

### 3.4 统一轨迹数据结构

无论英文还是中文，建议都转换为统一的中间数据格式，下游的平滑处理和鼠标控制模块只对接这一种结构，不关心数据来源：

```json
[
  {
    "stroke_id": 0,
    "points": [[x1, y1, t1], [x2, y2, t2], "..."],
    "pen_down": true
  }
]
```

- `points`：局部坐标系下的坐标序列，`x, y` 建议归一化到 0~1000 范围，`t` 为相对时间权重，用于后续生成速度曲线。
- `pen_down`：`true` = 落笔（鼠标按下状态移动），`false` = 提笔移动（跨字符/跨笔画的空中移动，鼠标松开状态移动）。

最终输出前，再统一做：整体排版（字与字之间的间距、签名整体的旋转倾斜角度）→ 缩放映射到用户框选的屏幕区域 → 生成真实时间戳的鼠标事件序列。

## 4. 自然手写感的关键算法点

- **贝塞尔曲线插值**：原始关键点之间用三次贝塞尔曲线（或 Catmull-Rom 样条）插值出更多中间点，避免折线感，曲线张力可调。
- **速度曲线**（决定鼠标移动的时间间隔，不是匀速 move）：直线段移动快、间隔短；转折、笔画起止处放慢、间隔变长，模拟真实运笔时手部加减速的过程。可以用关键点的曲率大小动态计算每段的移动延时。
- **起笔/收笔轻重感**：虽然普通鼠标轨迹无法表现压感粗细，但可以用起笔阶段更密集的采样点、收笔阶段轨迹微微外扬甩出的小尾巴（很多人手写签名收笔会有一个自然甩出的笔锋）来从视觉上模拟笔锋效果。
- **随机扰动（jitter）**：在每个点位置加入微小幅度的随机偏移（不破坏整体字形识别度），让多次生成的同一签名有细微差异，更像真人手写而非程序复制。扰动幅度需可配置，默认值要小，避免破坏可读性。
- **连笔判定**：根据相邻笔画（或相邻字）起止点的距离和书写习惯阈值，决定是"提笔重新落笔"还是"不提笔直接划过去"（连笔），中文姓名签名里这一点对真实感影响很大。

## 5. 鼠标自动化与平台适配

### 5.1 技术选型建议

|层|建议方案|说明|
|---|---|---|
|跨平台基础鼠标控制|Python + `pyautogui` 或 `pynput`|可以满足基本的 move/down/up，但默认的 moveTo 是匀速直线，需要自己写循环按计算出的时间间隔逐点移动，不能直接用库自带的插值|
|更精细的底层控制（可选）|macOS: `Quartz`/`CGEvent`（PyObjC）；Windows: `ctypes` 调用 `SendInput` / `mouse_event`|如果 pyautogui 的精度/时间控制粒度不够，可以分平台用更底层的系统 API，获得更精确的事件时间戳控制，体现速度变化|
|屏幕区域框选|自定义半透明全屏窗口（Tkinter / PyQt）|运行时弹出一个可拖拽框选的透明覆盖层，让用户画出签字区域，记录该区域在屏幕上的绝对像素坐标|

### 5.2 关键实现注意事项

- 鼠标 down → 沿轨迹连续 move → up 的过程中，move 的每一步都要有真实的时间间隔（用 `time.sleep` 控制），不能用支持库的"瞬移"或默认匀速接口，否则画出来还是机械感。
- Windows 和 macOS 的坐标系、DPI 缩放、多显示器情况下的坐标换算可能不同，需要做兼容处理，尤其 macOS Retina 屏的逻辑分辨率和物理像素换算。
- 跨字符/跨笔画的"提笔"移动，要先松开鼠标按键再移动到下一个起点，避免被识别成一笔拖到底的乱线。
- 执行前建议有 3 秒倒计时或快捷键触发机制，方便用户把鼠标焦点切换到目标软件的签字区域。

## 6. 用户使用流程（建议）

1. 启动程序，输入要签的姓名文字（支持中英文）。
2. 选择/预览生成的签名笔迹效果（可在程序内的预览画布看到笔迹轨迹示意图，不需要真的移动鼠标）。
3. 调整参数：签名大小、倾斜角度、书写速度、随机扰动强度。
4. 点击"开始签名"，程序提示用户在倒计时内切换到目标软件并框选/确认签字区域。
5. 倒计时结束后，程序自动接管鼠标，按生成的轨迹完成签名动作。
6. 签字过程中按 ESC 或预设快捷键可随时中断（安全考虑，避免鼠标失控误触其他区域）。

## 7. 安全与边界控制（务必实现）

- 签字执行期间必须有清晰的取消/中断快捷键（如 ESC），并且默认只允许在用户框选确认的矩形区域内移动鼠标，绝不允许超出该区域进行任何点击。
- 执行前必须有倒计时或二次确认，避免误触发导致鼠标在屏幕上乱画。
- 不做任何键盘记录、屏幕截图上传等无关功能，范围严格限定在"鼠标轨迹生成与回放"。

## 8. 分阶段交付建议（给 Codex 的开发顺序）

1. **阶段一**：用 Hershey 单线字体打通英文签名全流程——文字转轨迹点 → 贝塞尔平滑 → 速度曲线 → 屏幕区域框选 → 鼠标自动化执行。先验证整体架构可行。
2. **阶段二**：加入连笔判定、随机扰动、收笔甩锋等细节算法，提升英文签名的真实感。
3. **阶段三**：接入中文笔迹方案二（字体骨架提取）作为过渡，跑通中文流程。
4. **阶段四**：接入中文笔迹方案一（CASIA 真实笔迹库），替换骨架提取方案，提升中文签名质量到"真实手写"水准。
5. **阶段五**：打磨用户界面（输入框、参数调节、预览画布、框选交互），做成可直接使用的桌面小工具。

## 9. 待 Codex 在实现过程中自行调研确认的事项

- CASIA-OLHWDB 数据库的最新申请入口、是否仍在维护分发、数据格式解析的现成开源代码是否可用。
- Hershey Fonts 或同类单线字体在 npm/PyPI 上当前可用的、维护状态良好的封装库名称和用法。
- `pyautogui` / `pynput` 在最新版本下，在 macOS 上是否需要额外申请"辅助功能"（Accessibility）系统权限才能控制鼠标，并在程序启动时给出友好提示引导用户开启。
- Windows 上是否有权限或安全软件拦截模拟鼠标事件的情况，需要做兼容性测试。

## 10. 验收标准（建议）

- 英文姓名签名：轨迹流畅、有明显的连笔和速度变化，肉眼观感接近真实手写签名，而非匀速折线。
- 中文姓名签名：笔画顺序正确、字形可辨识，整体有连笔和自然速度变化，不是工整印刷体的逐笔复刻。
- 可在 macOS 和 Windows 上分别成功运行，并能在不同目标软件（如系统画图工具、PDF阅读器的签字控件）的指定区域内完成签名动作。
- 整个签字过程可被用户随时安全中断，不会出现鼠标失控点击到签字区域外的情况。
